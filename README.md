### 2026-03-14-19-00-00 · Safari 视频兼容性完整排查报告

#### 背景

网站首页 Hero Section 背景视频在 iPad Safari 上始终显示黑屏，其他所有内容正常。Safari 在视频无法播放时不报任何错误，只是静默黑屏。
缺乏真机（对的你们伟大的safari在13年前就不支持Windows了）的情况下需要尽可能测试可能的原因。

---
 
## 第一轮：Safari 视频播放通用兼容性修复
 
**假设：** Safari 对自动播放有更严格的策略，旧版 iOS 有额外要求。先排除 CDN 导致的格式传递错误。
 
**修复过程：**
 
- 重新设定 Cloudinary CDN 的自动解析方式，防止 CDN 推送 Safari 不支持的格式。本地测试 + 模拟器跑通，部署，真机测试后不成功。

  格式应该没问题了，猜想可能是播放时序的问题。Safari 加载视频比 Chrome 慢，如果在视频还没准备好的时候直接调用 `.play()`，Safari 会静默失败，什么都不说。
 
- 修复播放时序：改为等 `readyState >= 2` 再播放，否则监听 `canplay` 事件触发。本地 + 模拟器没问题，build，部署，真机测试后不成功。
 
  开始怀疑是不是本地测试机器本身的问题在干扰判断。为了排除这个可能性，补上旧版 iOS 的兼容适配，确认不是版本问题。
 
- 添加 `webkit-playsinline`，兼容早期 iOS Safari 莫名其妙要求的私有属性。本地 + 模拟器正常，build，部署，真机测试后没有效果。
 
  三步都没用。但无论如何不能让用户在失败的时候只看到黑屏，先加一个降级方案兜底。
 
- 添加 `preload="auto"` 和 `poster` 封面图，保证在完全崩溃的情况下宽屏设备至少有个静态画面，不是纯黑屏。
 
**结果：** 这个小时的尝试结论是完全无效的。
 
第一轮所有方向都走死了，寻求其他思路。问题可能不是播放逻辑，而是 Safari 和 CSS 渲染之间的问题。查文档，发现 Safari 在 CSS 渲染上有一个经典 bug。
 
---

## 第二轮：CSS Filter 渲染 Bug
 
**发现：** Safari 有一个有文档记录的已知缺陷——CSS `filter` 直接加在 `<video>` 元素上时，Safari 在创建硬件合成层时处理异常，导致画面全黑。视频本身其实在正常解码，只是 Safari 渲染层出了问题。Chrome 和 Firefox 完全不受影响。
 
**修复过程：**
 
- 将 `filter: brightness(95%) contrast(105%)` 从 `<video>` 元素移至外层 `<div>`，`<video>` 本身不附加任何 filter。本地 + 模拟器确认正常，build，部署，真机测试。
 
**结果：** 无效。
 
↓ 合成层的问题可能不只是 filter。也许 Safari 根本没有给这个视频元素分配独立的 GPU 渲染层。

---

## 第三轮：GPU 合成层 + 渲染 Fallback
 
**假设：** Safari 对 `absolute` 定位的 video 元素有时不会主动创建独立合成层，导致画面无法合成到屏幕上。那么我们需要尝试强制调用图形处理器来触发视频层的渲染。
 
**修复过程：**
 
- 给 `<video>` 加 `transform: translate3d(0,0,0)`，强制触发 GPU 合成层提升。这是处理此类 Safari 渲染问题的标准做法。
 
同时，如果 Safari 真的无法渲染视频，用户不应该只是看到黑屏什么都不知道。加一个超时保底。
 
- 添加 `timeupdate` 3 秒超时 Fallback：挂载后监听播放事件，若 3 秒内一直没有触发，自动切换为静态封面图背景，确保用户至少看到内容。
 
本地 + 模拟器测试正常，build，部署，真机测试。
 
Fallback方案没有触发，展示的是空白的时评控件，说明视频确实在解码，Safari 能收到视频流，但画面依然是黑的。
 
**结果：** 无效。本轮收获是排除了"视频根本没有在解码"的可能性。
 
代码层面能想到的方向基本都试过了。开始怀疑问题不在代码，而在 CDN 上的视频资产本身。

---

## 第四轮：更换 Cloudinary 视频资产
 
**假设：** Cloudinary 对这个特定文件的转码结果有问题，导致实际推送的视频格式不是预期的 H.264。那么可能是自动转换的问题，或者是传递上的问题。
 
**修复过程：**
 
- 重新导出视频，手动对视频进行 H.264 编码后上传至 Cloudinary，得到新的资产版本，替换页面中的视频 URL。本地 + 模拟器确认引用正确，build，部署，真机测试。
 
首次加载 autoplay 失效，确定问题排出后发现原因是 Cloudinary 对新上传的文件有按需转码延迟，视频还没处理完。系统强制刷新之后 autoplay 恢复正常。
 
但 iPad 的 Safari 在清理缓存之后依然黑屏。因为Safari并没有现代浏览器的自带调试方案，所以目前还是不确定到底为什么。
 
**结果：** 无效。
 
走投无路。代码没问题，CDN 资产换过了，所有能猜的方向都猜过了，Safari 依然不说话。
---

#### 第五轮：截图分析 + 根本原因确认

把 iPad Safari 的截图丢给Ai研究，显示页面只有 `<section>` 的 CSS 径向渐变背景（暗色 + 中央灰白光晕），`<video>` 元素完全透明。
猜想视频控件根本没有在运作，但是原来的视频资产已经被替换了，依旧无效，猜想是因为CDN的视频提供逻辑不同。

**根本原因：原始视频以 AV1 编码上传。**

AV1 在 Apple 设备上的支持情况：

| 设备 | 支持 AV1 |
|------|---------|
| iPhone 15+（A17 芯片） | 支持 |
| iPad Pro M4 | 支持 |
| 其他所有 iPhone / iPad | **不支持** |
| macOS Safari 17+ | 支持 |（就上个版本的macos）
| macOS Safari 16 及以下 | **不支持** |

也就是说，绝大多数正在被使用的 Apple 设备，根本无法播放这个视频。

我们在第一轮设置的 CDN 强制转码参数，理论上应该把 AV1 自动转成 H.264。它没有生效。Cloudinary 可能内部判断这个文件"已经足够优化"而跳过了转码。这是一个外部服务的不确定行为，我们无法预测，无法在开发环境中复现，也没有任何报错。

没辙了。
---

#### 第六轮（当前方案）：本地托管 + 双格式多源

**策略：** 脱离 Cloudinary 格式判断，将视频托管在 Hostinger VPS，为了保证读取的速度，同时为不同的浏览器提供 AV1 和 H.264 两个版本（因为h264视频会比较大，加载速度更慢。3.7Mb在现代互联网传递也是一个巨大的量。整个iphone17的官网也就这么大了）：

```
public/videos/
├── Herosection_av1.mp4   (1.9MB，Chrome/Firefox)
└── Herosection_h264.mp4  (3.7MB，Safari)
```

代码使用 `<source>` 多源，浏览器按 codec 支持自动选择：
- Safari → 跳过 AV1 → 选 H.264
- Chrome/Firefox → 选 AV1（体积小 ~50%）

### 2026-03-14-20-00-00 · 结案

**BrowserStack 真机测试通过。** iPad Safari 视频正常播放，问题完全解决。

根本原因确认为视频源文件以 **AV1 编码**上传，绝大多数 Apple 设备不支持 AV1 解码。最终方案：将 H.264 和 AV1 两个版本托管在 Hostinger VPS，通过 `<source>` 多源写法让浏览器按 codec 支持自动选择，Safari 选 H.264，Chrome/Firefox 选 AV1。

---



#### 结论

1. Safari 视频播放失败不报错，必须主动排查
2. AV1 在 Apple 生态覆盖率极低，面向消费者的视频必须提供 H.264 fallback
3. Cloudinary 的自动转码在实际使用中有不确定性，关键视频还是需要手动控制格式
4. `<source>` 多源写法是视频兼容性的最佳实践
